# Architecture Decisions

A short log of the load-bearing decisions made during Phases 0–6. When you (or a future you) wonder "why is it built like this?" — answers live here.

---

## 1. AI interprets, system structures, system enforces

State extraction from fiction is not deterministic; humans (or an LLM) read prose and infer state. But once a fact is extracted, **storage, comparison, and enforcement must be deterministic.** The system is reliable iff the deterministic layer is reliable, regardless of LLM weather.

Concretely:
- LLM produces `StateDelta` (probabilistic)
- Pydantic validates structure (deterministic)
- Conflict detector compares against store (deterministic)
- Merge mechanics, supersession, idempotency (deterministic)

---

## 2. One canon store, four namespaces, three confidence tiers

Reject the four-layer ontology (canon / character / plot / craft as separate stores) as premature. One flat store with namespaced keys + confidence tiers + content-hash retrieval gets 80% of the value at 5% of the complexity. Revisit only if the store exceeds ~5k entries.

| Confidence | Conflict behaviour |
|---|---|
| `hard_canon` | HIGH on disagreement — block merge |
| `event` | MEDIUM on disagreement — flag, allow merge |
| `inferred` | LOW on disagreement — auto-retcon |

This tiering is the single most important design choice. Without it, conflict detection either fires on every reinterpretation (too noisy) or never fires on real canon violations (too quiet).

---

## 3. Provenance is mandatory, not optional

Every `CanonEntry` carries `source_chapter` + `extracted_at_pass` + `created_at` + optional `superseded_by`. Without provenance, conflict reports are unactionable — the author has no idea where the original fact came from. With it, every conflict is one-click jumpable to its source.

---

## 4. Storage format: flat JSON, not SQLite

Single `canon_store.json` file. Git-diffable. Easy to inspect. Easy to back up. Fine to ~2k entries. SQLite if/when we cross that threshold.

The LLM cache *does* use SQLite — different access pattern (frequent random read by hash), different size profile.

---

## 5. Retrieval is deterministic, not semantic

Entity detection is a word-boundary regex scan against `entity + aliases` in the store. No embeddings, no LLM. Reasons:

- **Reproducibility** — same chapter + same store = same slice, always
- **Speed** — sub-millisecond
- **Debuggability** — when retrieval misses, fix by adding an alias

Trade-off: pronoun resolution doesn't work. Accept that in v1; revisit only if analysis quality suffers measurably.

---

## 6. Two-layer cache: L1 (local) and L2 (provider)

- **L1** saves on **repeat runs** (100% saving on hits)
- **L2** saves on **first runs with large stable context** (90% saving on cached prefix)

Both are needed. The two cost shapes are different.

L1 cache key is a content hash of every input that affects output. Any change to any input → automatic miss. Manual invalidation is a separate API for event-driven cases.

L2 puts the manuscript (~106k tokens) in the cached prefix; canon slice + chapter pointer in the per-call payload. Manuscript is stable across all chapters, so the cache stays warm through a back-to-back batch.

---

## 7. Cache invalidation strategy (all four triggers)

Per the user's spec:

- **TTL** — optional per-entry expiry; `sweep_expired()` job
- **Content-hash mutation** — input change → key change → automatic miss (old entries become unreachable)
- **Manual / event** — `cache.invalidate(key)` and `cache.clear()`
- **LRU eviction** — when total size > 1 GiB, evict least-recently-accessed

---

## 8. Provider abstraction by tier, not by name

`roles.py` maps role → tier. `tiers.py` maps tier → `provider:model_id`. The pipeline code only knows the role. Model names live in one file. Swap models without touching any pipeline code.

Today's mapping (May 2026, dated):
- T2_premium (default for continuity extraction) → `anthropic:claude-opus-4-7`
- T1_default (cost-conscious workhorse) → `google:gemini-3-pro` (stub)
- T3_fast (cheap iteration) → `google:gemini-3-flash` (stub)
- T4_reasoning (heavy logic) → `openai:gpt-5.5-thinking` (stub)

---

## 9. Idempotency via pass_id

Every entry's `extracted_at_pass` field tells you which run wrote it. If a `pass_id` has already produced entries in the store, the pipeline treats the run as already-applied and returns `SKIPPED_IDEMPOTENT`. Re-runs are safe. Interrupted jobs are safe to retry.

To force a fresh extraction: use a new `pass_id` (auto-generated by default with UTC timestamp).

---

## 10. Supersession over deletion

Nothing is ever deleted. Retcon = create a new entry + point the old at it via `superseded_by`. History is preserved; queries default to active entries; `include_superseded=True` reveals the timeline.

This matters more for fiction than for software. Authors change their minds; readers don't see the changes; the author needs to remember what the previous reading was and why it was retired.

---

## 11. Conflict detection: false negatives over false positives

Value comparison normalises (lowercase, whitespace, light punctuation). Intentionally simple. **False negatives are recoverable** (you'll catch the miss in review). **False positives erode trust** — once the author starts skipping reports because they're noisy, the system is dead.

What we DON'T detect (yet):
- Cross-ID semantic contradictions
- Orphan entity references
- Timeline ordering across multiple events

The extractor prompt instructs the model to reuse existing IDs, so cross-ID drift should be rare. v2 may add LLM-assisted compare if it isn't.

---

## 12. Author UX: markdown files in a folder

Pending conflict reviews are written to `pending/ch<N>__<pass_id>.md`. Author reads in their editor. Decides. Acts. Re-runs.

Could be a web UI later. Markdown-in-folder is the right MVP — it's git-friendly, lossless, scriptable, and requires zero new infrastructure.

---

## 13. The seed is ground truth from the manuscript, not from planning docs

The first version of `qs_seed.py` imported terminology from the batch execution docs (`Fade`, `A&S mug`, `D1 mechanism`). None of these appear in the actual manuscript prose. The polish pass (Phase 6.5) rewrote the seed from manuscript audit — every claim is now verifiable with grep.

**Rule:** if you add a seed entry, you should be able to point to a specific manuscript passage that establishes it. If you can't, it doesn't belong as `hard_canon` — at most `inferred`.

---

## 14. Polish before live API

Before spending a cent on the Galaxy proxy:

- Verified all named characters appear in the manuscript text
- Verified all world-building terms (Bleed, ICS, Sphere, Threshold, SEP-NULL) are real
- Fixed two wrong chapter titles in plot anchors (Ch 10, Ch 12)
- Removed three invented concepts (Fade, A&S mug, D1 mechanism)
- Documented the two-Mirror-Hunt-chapters ambiguity as a craft note
- Added Marcus Chen, Varn, the Replacement Anomaly pattern, the SHADOW/DELTA-14 file

Cost of polish: zero API spend. Value of polish: every extraction pass starts from accurate canon, not from my hallucinations. Worth it.

## 15 — Audit trail loss (May 2026)

Between the initial Phase 6 extraction work and the Ch 1–4 backfill, the
extractor passed an `extracted_at_pass` value to the LLM as part of the
prompt instruction but did not (a) substitute the literal pass_id into
the prompt template, and (b) defensively override LLM output. The LLM
was free to invent its own past-dated pass IDs (e.g. `ch0_2025-07-08T12:00:00`).

Approximately 206 canon entries ended up with fabricated `extracted_at_pass` 
values, primarily across multiple independent Prologue extraction runs 
and stray runs for other chapters.

**Remediation:**
1. The bug was fixed via prompt-template substitution and defensive override
   in `extractor.py` (May 2026).
2. A regression test (`tests/test_extractor_pass_id.py`) prevents recurrence.
3. The 206 contaminated entries were re-tagged with `extracted_at_pass = "audit_trail_lost"`
   to make the contamination explicit and audit-script-friendly.

**Why we did not re-extract from scratch:**
The contaminated entries' *content* was verified-good via grep checks against the
manuscript. Only the provenance metadata was wrong. Re-extracting would have cost
additional API calls and risked materially different downstream extractions due to
canon-slice drift. The cost of explicit data loss was lower than the cost of
re-running.

## 16 — Seed identity-attribution error (May 2026)

Two seed entries (`bell.identity` and `bell.discovery`) incorrectly stated
that Bell discovers *Kain's* hidden ICS readings in Ch 8. The actual
manuscript clearly shows Bell discovers *Hayden's* ICS readings (her own
partner's). This is a content error introduced during the Phase 6.5 seed
polish, where I read manuscript grep fragments out of context.

**How it was caught:** Ch 8 backfill produced 22 entries about the Hayden
data — correct content — but did NOT supersede the two wrong seed entries.
The extractor invented new ids (`bell.pov_chapter`, `hayden.ics_full_trajectory`)
rather than reusing the existing wrong-but-collisional ids. Result:
silent parallel canon — three Ch-8-anchored Bell entries, two saying Kain,
one saying Hayden, no conflict report.

**Remediation:**
1. Manually superseded `bell.identity` -> `bell.identity.v2` and
   `bell.discovery` -> `bell.discovery.v2` with corrected attribution.
   The v2 entries are deliberately brief and point at the Ch 8 entries
   that cover the full story.
2. Updated `extract_delta.txt` (v1.1 -> v1.2) with an explicit
   "Identity attribution check" rule directing the extractor to reuse
   existing ids on identity corrections.
3. Bumped `extractor.v2` -> `extractor.v3` in `cache_key_parts` to force
   cache miss on future extractions.
4. Added regression test
   `tests/test_extractor_identity_routing.py::test_extractor_can_correct_identity_attribution`
   verifying the system accepts identity-correction entries through
   existing ids.

**Why we did not re-extract Ch 8:**
The Ch 8 content is correct and rich (22 entries, all manuscript-grounded).
Re-extracting risks losing this content for a marginal benefit. The
parallel-canon issue is resolved by retiring the wrong seed entries,
which is what the supersession accomplishes.

**Open question for the next backfill cycle:**
Are there other seed entries with similar identity errors that haven't
yet been exposed by chapter extraction? Likely candidates: the Reyes
entries (case vs M. Reyes), the Aspect entry (claims it's a designation
but with no manuscript grounding for the specifics). Watch supersession
patterns in remaining chapters for similar silent-parallel-canon shape.
